# 核心算法说明

本文档将数据查询系统的核心算法从代码实现中抽象出来，用数学语言和伪代码描述。

## 一、问题定义

给定一个三维数组 `D[t, y, x]`，其中：

- `t ∈ [0, T)` 为时间维度
- `(y, x)` 为空间平面上的像素坐标，共 `P = H × W` 个像素

用户指定：

- 一组选中像素集合 `S ⊆ {0, 1, ..., P-1}`（扁平索引）
- 一个时间范围 `[t_start, t_stop)`
- 一个最大显示点数 `N_display`

要求计算区域均值随时间的变化曲线：

```
mean(t) = Σ D[t, p] / |S|,  对所有 p ∈ S
```

当时间跨度远大于显示点数时，需要将时间轴分桶聚合，每桶输出一个均值。

## 二、核心挑战

在最大规模下（`P = 200万`，`T = 50000`），单类数据体积可达数百 GiB。直接遍历所有选中像素的所有时间帧，I/O 量和计算量都不可接受。

核心目标是：**预处理一次，后续任意选区 × 任意时间范围的查询都能秒级响应。**

## 三、算法一：空间块分解

### 思路

将 `P` 个像素按固定块大小 `B` 分成 `⌈P/B⌉` 个空间块。预处理时，为每个块预先计算每个时间帧的像素值总和：

```
block_sums[t, k] = Σ D[t, p],  对所有 p ∈ Block_k
```

查询时，将选区 `S` 拆分为两部分：

1. **完整块** `F`：块内所有像素都被选中
2. **非完整块像素** `R`：只选中了部分像素的那些零散像素

```
total(t) = Σ block_sums[t, k]  (k ∈ F)
         + Σ D[t, p]           (p ∈ R)

mean(t) = total(t) / |S|
```

### 复杂度分析

设 `|F|` 为完整块数，`|R|` 为零散像素数。

- 不用此算法：每帧读取 `|S|` 个像素 → I/O 量 `O(T × |S|)`
- 用此算法：每帧读取 `|F|` 个块汇总值 + `|R|` 个像素 → I/O 量 `O(T × (|F| + |R|))`

当选区是大面积连续区域时，`|F|` 远小于对应像素数，`|R|` 只集中在选区边缘，加速比显著。

### 伪代码

```
function selection_plan(S, block_size, pixel_count):
    对每个像素 p ∈ S:
        block_id = p ÷ block_size
        统计每个 block 被选中的像素数 counts[block_id]

    block_lengths[k] = Block_k 中的实际像素数（最后一块可能不满）

    full_blocks = { k | counts[k] == block_lengths[k] }
    partial_pixels = { p ∈ S | p 所在块不在 full_blocks 中 }

    return (full_blocks, partial_pixels)
```

## 四、算法二：Partial Ranges 连续区间合并

### 思路

非完整块的零散像素索引是一个排序后的整数数组。直接用这些索引做 fancy indexing（散点读取）时，每个索引触发一次独立的内存/磁盘访问，效率低下。

优化方法：将排序后的像素索引合并为连续区间列表 `[(start₀, stop₀), (start₁, stop₁), ...]`，查询时使用切片索引 `D[t, startᵢ:stopᵢ]` 读取，走连续内存路径。

### 合并算法

```
function indices_to_ranges(indices):
    // indices 已排序去重
    breaks = 所有满足 indices[i+1] - indices[i] ≠ 1 的位置
    每两个相邻 break 之间形成一个连续区间 [indices[start], indices[end] + 1)
    return 区间列表
```

### 示例

```
indices = [3, 4, 5, 10, 11, 20]

breaks 在位置 2 (5→10) 和 4 (11→20)

ranges = [(3, 6), (10, 12), (20, 21)]
```

查询时：

```
total = 0
for (rs, re) in ranges:
    total += sum(D[t_start:t_stop, rs:re])
```

每个切片读取的是连续内存，对 mmap 文件非常友好。

## 五、算法三：时间 LOD 金字塔

### 思路

当用户查询 `T = 50000` 帧但只显示 `N = 500` 个点时，每个显示点对应 100 帧的聚合。直接从原始 `block_sums` 读取 50000 行再聚合，I/O 量仍然很大。

解决方法：预处理时构建多级时间汇总（Level of Detail），每一级将时间分辨率减半：

```
LOD Level 0: block_sums[t, k]                       — T 帧（原始）
LOD Level 1: lod₁[j, k] = Σ block_sums[2j:2j+2, k] — ⌈T/2⌉ 帧
LOD Level 2: lod₂[j, k] = Σ lod₁[2j:2j+2, k]      — ⌈T/4⌉ 帧
  ...
LOD Level N: 直到帧数 ≤ MIN_DISPLAY_POINTS (500)
```

LOD Level `k` 的第 `j` 帧存储的是原始帧 `[j·2ᵏ, min((j+1)·2ᵏ, T))` 的**求和值**（不是均值，以保证可组合性）。

### 查询时的级别选择

```
function choose_lod_level(span, max_points, max_level):
    level = 0
    while level < max_level:
        next_frames = ⌈span / 2^(level+1)⌉
        if next_frames < max(500, max_points):
            break  // 再粗一级就不够显示点数了
        level += 1
    return level
```

### 查询流程（聚合模式，LOD 加速）

```
stride = 2^level
lod_start = t_start ÷ stride
lod_stop  = ⌈t_stop / stride⌉

// 1. 完整块：从 LOD 级别直接读取
block_totals = lod[level][lod_start:lod_stop, full_blocks].sum(axis=1)

// 2. 边界修正：LOD 帧边界可能超出 [t_start, t_stop)
if lod_start * stride < t_start:
    从 Level 0 减去 [lod_start*stride, t_start) 的多余部分
if lod_stop * stride > t_stop:
    从 Level 0 减去 [t_stop, lod_stop*stride) 的多余部分

// 3. 非完整块像素：仍从原始 cube 按 LOD 帧边界分组读取
for 每个 LOD 帧 j:
    orig_start = max(t_start, (lod_start + j) * stride)
    orig_stop  = min(t_stop,  (lod_start + j + 1) * stride)
    pixel_totals[j] = Σ D[orig_start:orig_stop, ranges] （使用 partial ranges）

// 4. 组合
for 每个 LOD 帧 j:
    count = 该帧覆盖的实际原始帧数
    display_values[j] = (block_totals[j] + pixel_totals[j]) / (|S| × count)
```

### 关键性质

- LOD 存储的是**求和值**，不是均值。这保证了任意两个相邻帧可以直接相加合成上一级，无需知道各帧的原始帧数。
- 最后一帧可能覆盖不足 `2ᵏ` 个原始帧，但求和值仍然正确。查询时通过实际帧数 `count` 来计算均值。

### 存储开销

```
LOD 总行数 = T/2 + T/4 + T/8 + ... ≈ T
LOD 文件大小 ≈ block_sums 大小 × 50%（因为 Level 0 不重复存储）
```

在最大规模下（T=50000，489 块），每类 LOD 约 93 MiB，相比主数据体积可忽略。

### I/O 对比

查询 50000 帧、显示 500 点、完整块查询时：

| 方法 | 读取行数 | 读取量（489 块） |
|---|---|---|
| 无 LOD（逐帧聚合） | 50000 行 | ~186 MiB |
| LOD Level ≈6（stride=64） | ~782 行 + 边界修正 | ~3 MiB |

## 六、算法四：多层并行

### 层次结构

```
┌─────────────────────────────────────────────┐
│  Level 1: 跨类别并行                         │
│  ThreadPool(N_categories)                    │
│  ┌───────────────────────────────────────┐   │
│  │  Level 2: 单类别内时间块并行            │   │
│  │  ThreadPool(workers / N_categories)   │   │
│  │  每个线程处理独立的时间切片             │   │
│  └───────────────────────────────────────┘   │
└─────────────────────────────────────────────┘
```

### 为什么用线程而非进程

查询阶段的核心操作是 NumPy 数组运算（`sum`、切片）和 mmap 文件读取。这两类操作都会释放 Python GIL：

- NumPy 的 C 层计算在执行期间释放 GIL
- mmap 的页面读取是内核级 I/O，不持有 GIL

因此 `ThreadPoolExecutor` 可以获得真正的并行加速，同时避免了多进程序列化 mmap 句柄的开销。

### 线程分配策略

```
总线程数 = workers（默认 min(16, CPU 核心数)）

跨类别查询时:
    cat_workers   = min(类别数, workers)
    inner_workers = workers ÷ cat_workers

    每个类别分配 inner_workers 个线程处理其时间块
```

## 七、算法组合：一次查询的完整流程

```
输入: 选区掩码 mask[H, W], 时间范围 [t_start, t_stop), 显示点数 N

1. mask → flat_indices                          // np.flatnonzero，微秒级
2. flat_indices → SelectionPlan                 // 空间块分解，结果缓存
     ├── full_blocks: [k₁, k₂, ...]
     ├── partial_ranges: [(s₁,e₁), (s₂,e₂), ...]
     └── partial_pixel_count: int

3. 判断是否需要聚合:
     span = t_stop - t_start
     if span > N: 进入聚合模式

4a. 聚合模式 + LOD 可用 (span >> N):
     ├── 选择 LOD 级别 → stride = 2^level
     ├── 完整块: 从 LOD 读取 ~N 行 block_sums
     ├── 边界修正: 从 Level 0 减去超出部分
     ├── 非完整块: 从 cube_tyx 按 LOD 帧边界分组读取（多线程并行）
     └── 组合: display_values[j] = total[j] / (|S| × count[j])

4b. 聚合模式、无 LOD:
     ├── 按显示点数线性分桶（linspace）
     ├── 每桶内按时间块流式求和（多线程并行）
     └── display_values[j] = bin_total[j] / (|S| × bin_size[j])

4c. 非聚合模式 (span ≤ N):
     ├── 按时间块读取，每块内对选区求和（多线程并行）
     └── raw_values[t] = frame_sum[t] / |S|

5. 多类别时，步骤 3-4 对所有类别并行执行（线程池）

6. 返回 (display_time[], display_values[]) 用于绑图
```

## 八、预处理流程

```
输入: 原始 3D 数组 source[...], 轴顺序 layout, 块大小 B

1. 根据 layout 识别 (T, H, W) 维度
2. 创建 cube_tyx.npy (T, H, W) 和 block_sums.npy (T, ⌈P/B⌉) 的 memmap
3. 按时间块流式处理（多线程并行写入 memmap 的不重叠区域）:
     for 每个时间块 [t_start, t_stop):
         chunk = 从 source 按 layout 切片并转置为 (t, y, x)
         cube_tyx[t_start:t_stop] = chunk
         block_sums[t_start:t_stop] = 按空间块求和

4. 构建 LOD 金字塔（可选，默认启用）:
     current = block_sums (T 行)
     while current.行数 > 500:
         next_level = 相邻两行求和（奇数行末尾直接保留）
         保存 next_level
         current = next_level
     合并所有级别 → block_sums_lod.npy

5. 生成空间预览图 preview.npy（抽样若干时间帧取均值）
6. 写入 meta.json（含 LOD 级别数和偏移表）
```

## 九、算法特性总结

| 特性 | 说明 |
|---|---|
| 预处理一次，查询多次 | 所有加速结构（block_sums、LOD）在预处理阶段生成 |
| 选区无关的预处理 | 预处理不依赖选区，任意选区都能利用已有结构 |
| LOD 存储求和值 | 保证可组合性：任意相邻帧可直接合成上一级 |
| Partial ranges 切片读取 | 零散像素合并为连续区间，走内存连续路径 |
| 线程安全的并行 | mmap 只读 + 输出数组不重叠写入，无需锁 |
| 向后兼容 | 无 LOD 数据时自动回退到逐帧聚合路径 |
| 所有参数可配置 | `workers`、`block_size`、`--no-lod` 均可调整 |
