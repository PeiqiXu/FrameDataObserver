# 代码文件解读

本文档用于说明当前目录下主要代码文件分别在做什么，以及它们是如何对应《数据查询要求》中的需求的。

## 一、项目整体思路

这个项目的实现核心不是“每次查询时直接从原始 3D 大数组里全量扫描”，而是采用了两阶段方案：

1. 预处理阶段：把原始数据转换成更适合查询的存储形式。
2. 查询阶段：基于预处理结果，只读取需要的时间段和像素区域，并在必要时做显示级聚合。

这样做的目标，是尽量满足需求文档中的这条要求：

- 首次加载后，无论调整时间范围还是调整选区，都尽量做到秒级响应。

项目当前主要依赖以下几个技术点实现这个目标：

- 内存映射加载 `npy` 文件，避免一次性整段读入
- 把原始数据统一转换为时间主序 `cube_tyx.npy`
- 额外生成空间块汇总 `block_sums.npy`
- 对超长时间范围自动做显示点聚合
- 对选区拆分出”完整块”和”零散像素”，分别走不同计算路径
- 多线程并行：跨类别查询并行 + 单类别时间块并行 + 预处理并行，充分利用多核 CPU

## 二、代码文件分别做什么

### 1. `query_engine.py`

这是整个项目最核心的文件，负责两大部分能力：

1. 预处理原始数据
2. 提供查询引擎

它的主要职责包括：

- 定义支持的原始数据轴顺序：
  - `xyt`
  - `yxt`
  - `tyx`
  - `txy`
- 将原始 3D 数组统一转换为标准格式 `cube_tyx.npy`
- 生成按空间块汇总后的 `block_sums.npy`
- 生成时间 LOD 金字塔 `block_sums_lod.npy`（可通过 `lod=False` 或 `--no-lod` 禁用）
- 生成预览图数据 `preview.npy`
- 生成描述 prepared 数据集的 `meta.json`（包含 LOD 级别数和偏移表）
- 提供 `QueryEngine.query_category()` 用于按选区和时间范围返回区域均值曲线
- 提供 `QueryEngine.query_categories()` 用于多类别并行查询
- 预处理和查询均支持 `workers` 参数控制并行线程数

#### 它如何满足需求

对应需求“原始数据是 3 维数组，且可能有不同轴顺序”：

- 通过 `LAYOUT_AXES` 和 `_slice_time_chunk()` 支持多种输入布局
- 预处理时统一变换为时间主序格式，后续查询就不再关心原始布局

对应需求“任意选定若干像素点，计算均值曲线”：

- 选区最终会转换成扁平像素索引
- 查询时读取这些索引对应的值并按时间求平均

对应需求“时间跨度很大可以抽样显示，显示点不少于 500”：

- 文件中定义了：
  - `MIN_DISPLAY_POINTS = 500`
  - `DEFAULT_DISPLAY_POINTS = 1200`
- 当查询时间跨度大于显示点数时，自动按时间桶聚合，而不是返回完整时间序列

对应需求”首次加载后尽量秒级响应”：

- 用 `np.load(..., mmap_mode=”r”)` 做内存映射
- 用 `block_sums.npy` 加速大片连续区域查询
- 用分块时间窗口方式做流式统计，避免构造超大中间数组
- 对选区分析结果做缓存，避免同一选区重复拆分
- 非完整块像素按连续区间合并（partial ranges），使用切片读取代替逐像素 fancy indexing
- 时间 LOD 金字塔：大时间跨度聚合查询时，直接从预计算的粗粒度 LOD 级别读取 block_sums，跳过大量原始帧
- 使用 `ThreadPoolExecutor` 多线程并行处理时间块，充分利用多核 CPU
- 多类别查询通过 `query_categories()` 并发执行，线程自动在类别间和时间块间分配

#### 关键设计

**选区拆分 + partial ranges 优化**

它把选区拆成两部分：

- 完整块：该块内所有像素都被选中
- 非完整块（partial ranges）：只选中了该块中的一部分像素

查询时：

- 对完整块，直接从 `block_sums.npy`（或 LOD 级别）读取块汇总值
- 对非完整块，零散的像素索引会被合并为连续区间 `[start, stop)` 列表，然后使用切片索引 `flat_cube[t, start:stop]` 从 `cube_tyx.npy` 读取

这种做法兼顾了两种情况：

- 大面积连续选区时，速度更快
- 小范围或稀疏选区时，仍然保持精确
- 非完整块的切片读取比逐像素 fancy indexing 快数量级（连续内存 vs 散点访问）

**时间 LOD 金字塔**

预处理阶段为 `block_sums` 构建多级时间汇总：

- Level 0 = 原始数据（T 帧），Level 1 = 每 2 帧求和（T/2 帧），Level 2 = 每 4 帧（T/4 帧）…直到帧数降至 500 以下
- 所有级别合并存储在 `block_sums_lod.npy`，通过 mmap 和偏移表按需读取
- 查询时自动选择最佳级别：`_choose_lod_level()` 找到最粗但仍 >= 显示点数的级别
- LOD 帧边界上首尾不对齐的部分，从 Level 0 精确修正
- LOD 默认启用，可通过 `--no-lod` 或 `lod=False` 禁用
- 存储开销约为 `block_sums.npy` 的 50%，相比主数据体积可忽略

### 2. `prepare_data.py`

这是预处理命令行入口文件。

它本身不负责核心计算，而是把 `query_engine.py` 中的 `prepare_category()` 封装成一个可直接执行的命令行程序。

它的职责包括：

- 接收命令行参数，例如：
  - 原始文件路径
  - prepared 输出目录
  - 类别名称
  - 原始轴顺序
  - 保存 dtype
  - 空间块大小
  - 预览抽样帧数
  - 并行线程数（`--workers`）
  - 是否禁用 LOD（`--no-lod`，默认启用）
- 调用 `prepare_category()` 生成 prepared 数据
- 在控制台打印预处理结果摘要

#### 它如何满足需求

需求中要求原始数据能够先准备成适合快速查询的结构，这个文件就是入口。

如果没有这个步骤，查询阶段就只能直接面对原始大体积数据，性能很难达到要求。

因此可以把它理解为：

- 把“原始数据”转换为“可快速查询的数据”

### 3. `data_query_app.py`

这是桌面交互界面，负责把查询能力变成一个可操作的应用。

它使用：

- `matplotlib`
- `matplotlib.widgets`
- Tk 后端或 macOS 后端

这个文件主要负责：

- 加载 prepared 类别数据
- 显示空间预览图
- 让用户在空间平面上选择像素
- 输入时间范围和显示点数
- 发起查询（多类别通过 `query_categories()` 并行执行）
- 将多个类别的结果绘制在同一张曲线上

#### 当前界面支持的操作

- 单点点击切换像素选中状态
- 矩形批量添加选区
- 矩形批量移除选区
- 清空选区
- 选择一个或多个类别同时查询
- 输入时间起止范围
- 输入最大显示点数

#### 它如何满足需求

对应需求“在 X*Y 空间平面上任意选择若干像素点”：

- 通过点选和矩形增删的方式实现选区编辑
- 最终将选区维护为 2D 布尔掩码 `selection_mask`

对应需求”绘制给定时间范围内区域均值随时间变化曲线”：

- 点击 `Query` 后，界面会将 `selection_mask` 转换为扁平索引
- 然后通过 `QueryEngine.query_categories()` 对所有被勾选的类别并行查询
- 最后把各类别曲线绘制出来

对应需求“3 大类独立计算并绘图”：

- 每个类别都对应一个 prepared 数据目录
- 选区相同，但每个类别单独查询、单独绘制

#### 需要注意的一点

`README.md` 中提到界面支持：

- 导入 `.npy` 掩码
- 导入 `x,y` 两列坐标 CSV

但从当前源码实现来看，界面里还没有看到对应的导入入口或文件读取逻辑。

也就是说，当前实际已经落地的选区方式主要是：

- 点选
- 矩形添加
- 矩形移除

这不影响核心需求的实现，因为需求本身只要求“任意选定若干像素点”，当前界面已经能做到这一点；只是 README 里提到的“外部文件导入选区”还没有在界面中真正实现。

### 4. `generate_demo_data.py`

这是演示数据生成脚本。

它的作用是：

- 生成 3 类模拟 3D 数据
- 保存到 `demo/raw/`
- 再立即调用 `prepare_category()` 做预处理
- 生成 `demo/prepared/`
- 使用 `ProcessPoolExecutor` 让 3 个类别的生成和预处理并行执行

这样用户不需要先准备自己的真实数据，也可以直接跑通演示流程。

#### 它做了什么

- 构造带有时间波动、热点移动、条纹和噪声的模拟数据
- 分别生成：
  - `class_a`
  - `class_b`
  - `class_c`
- 三类分别使用不同 dtype：
  - `float32`
  - `int16`
  - `int8`

#### 它如何满足需求

需求中说原始数据分 3 大类，每类独立计算。

这个文件就是用来模拟这种实际场景的，使整套流程可以直接演示：

- 3 类数据分别预处理
- 用相同选区对 3 类数据分别查询
- 最终分别绘图

此外，它还顺带验证了系统对不同精度数据类型的适配能力。

### 5. `test_query_engine.py`

这是单元测试文件，用于验证查询引擎核心逻辑是否正确。

它当前主要覆盖了 6 类测试：

1. `tyx` 输入布局下，查询结果与暴力计算一致
2. `xyt` 输入布局下，预处理转换正确
3. 超长时间范围查询时，会自动聚合显示且不返回完整原始序列
4. `dtype=auto` 时，会保留原始数据类型并得到正确结果
5. 时间 LOD 加速查询结果与暴力计算一致（含完整块 + 非完整块混合选区）
6. LOD 元数据和文件正确生成

#### 它如何满足需求

它并不是直接实现业务功能，但它能帮助证明：

- 曲线数值是对的
- 不同输入布局是兼容的
- 大时间跨度聚合策略按预期工作
- 混合 dtype 的类别处理逻辑是成立的
- LOD 加速路径的计算结果与原始路径一致

这对“需求是否真正被正确实现”非常重要。

### 6. `benchmark_max_dataset.py`

这是性能评估和资源估算脚本，不属于主业务流程，但对验证设计是否合理很有帮助。

它主要做两类 benchmark：

1. `prepare_stream`
   - 测试预处理阶段按时间块流式生成 `block_sums.npy` 的表现
2. `query`
   - 测试在大规模条件下，按完整块查询时的耗时和内存表现

#### 它如何满足需求

需求文档给出了非常大的数据上限：

- `X * Y` 最大约 200 万像素
- `T` 最大 50000

这个文件的意义在于：

- 不只是“功能上能算”
- 而是进一步验证在大规模下，这种架构是否还有现实可行性

它是对项目性能设计的辅助证明。

## 三、非代码但重要的文件

### 1. `README.md`

这是项目说明文档，主要包含：

- 项目目标
- 文件说明
- 使用方式
- 性能设计
- 资源估算
- 已知边界
- 测试方法

它可以看作这套实现的整体设计说明书。

### 2. `demo/prepared/class_*/meta.json`

这些不是源码，而是预处理后的元数据文件。

它们记录：

- 类别名称
- cube 文件名
- block_sums 文件名
- preview 文件名
- 原始数据路径
- 原始布局
- 尺寸信息
- 块大小和块数量
- LOD 级别数（`lod_levels`）
- LOD 文件名（`block_sums_lod_file`）
- LOD 偏移表（`lod_offsets`，用于从合并文件中定位各级别的行范围）

查询界面和查询引擎会依赖这些元数据来正确加载 prepared 数据集。旧版不含 LOD 字段的 `meta.json` 也能正常加载（向后兼容，`lod_levels` 默认为 0）。

## 四、从需求到实现的对应关系

下面按需求文档中的核心要求，对应到当前代码实现：

### 1. 原始数据分 3 大类，每类独立计算

对应实现：

- `generate_demo_data.py` 生成 3 类示例数据
- `prepare_data.py` / `query_engine.py` 支持每类独立预处理
- `data_query_app.py` 支持同时勾选多个类别分别查询绘图

### 2. 单类原始数据是一个 3 维数组

对应实现：

- `query_engine.py` 中 `prepare_category()` 要求输入是 3D `.npy`
- 并支持多种轴顺序转换

### 3. 在空间平面上任意选定若干像素点，计算区域均值

对应实现：

- `data_query_app.py` 负责构建选区
- `mask_to_flat_indices()` 将 2D 掩码转为扁平索引
- `QueryEngine.query_category()` 负责对选中像素按时间求均值

### 4. 支持任意时间范围查询

对应实现：

- `data_query_app.py` 提供 `start` / `stop` 输入框
- `QueryEngine.query_category()` 只处理指定时间窗口

### 5. 时间跨度大时允许抽样显示，且不少于 500 个点

对应实现：

- `query_engine.py` 中定义最小显示点数 500
- 查询跨度过大时按时间桶聚合
- 当存在 LOD 金字塔时，自动选择最优级别，完整块的汇总值直接从 LOD 读取（如 50000 帧 → 只读 ~500 行），大幅减少 I/O

### 6. 首次加载后，调整选区或时间范围尽量秒级响应

对应实现：

- prepared 数据预处理
- 内存映射读取
- 空间块汇总
- 查询分块流式聚合
- 选区规划缓存
- 非完整块像素按连续区间合并（partial ranges），切片读取代替逐像素散点访问
- 时间 LOD 金字塔加速大跨度聚合查询（完整块从 LOD 直接读取 ~500 行，默认启用，可 `--no-lod` 禁用）
- 多线程并行加速（四个层面）：
  - 跨类别查询并行：`query_categories()` 同时查询多个类别
  - 单类别时间块并行：`query_category()` 内时间块由线程池并发处理
  - 跨类别预处理并行：`generate_demo_data.py` 中多进程同时预处理
  - 单类别预处理并行：`prepare_category()` 内时间块由线程池并发写入 memmap

这也是当前架构最核心的设计目标。

## 五、可以怎样理解这套架构

如果用一句话概括：

- `prepare_data.py` 和 `query_engine.py` 先把数据整理成”可快速查询的索引化结构”（预处理支持多线程并行写入）
- `data_query_app.py` 提供选区和查询界面（多类别查询并行执行）
- `QueryEngine` 在查询时结合 `cube_tyx.npy`、`block_sums.npy` 和 LOD 金字塔做精确且尽量高效的均值计算，并通过线程池对时间块并行处理
- `generate_demo_data.py` 负责搭建演示环境（多类别多进程并行生成）
- `test_query_engine.py` 负责验证正确性
- `benchmark_max_dataset.py` 负责评估大规模下的资源与性能

因此，这个项目的本质是：

- 一个面向大规模时空 3D 数据的区域时间序列查询原型，支持多核并行加速

## 六、当前结论

从当前仓库代码来看，核心需求已经基本覆盖：

- 支持 3D 数据预处理（多线程并行写入）
- 支持多种原始轴顺序
- 支持空间选区（非完整块像素自动合并为连续区间，优化读取性能）
- 支持时间范围查询（时间块多线程并行处理）
- 支持大时间跨度自动聚合显示（时间 LOD 金字塔加速，默认启用，可 `--no-lod` 禁用）
- 支持多类别独立绘图（跨类别并行查询）
- 预处理和查询均支持 `workers` 参数控制并行度，默认 `min(16, CPU 核心数)`

并且在实现上已经考虑了大规模数据场景下的性能问题，通过四层并行 + LOD 金字塔 + partial ranges 切片优化充分利用硬件。

当前最明显的一个“文档与实现存在差异”的点是：

- README 提到了导入掩码和 CSV 坐标
- 但界面源码中尚未看到对应的导入功能入口

除此之外，整体代码结构是比较清晰的，职责划分也比较明确。
